## Clients & infrastructure: macOS app, web, CLI, gateway

The platform ships the assistant across four independent surfaces — a native macOS app, two web shells, an MV3 Chrome extension, and a `max` CLI — all fronted by a single Bun/TypeScript **Gateway** that is the only thing on the public network. Everything below is platform code authored by Vellum/Max AI. Where the author's separate work (the kimi-agent provider, claude-subscription hardening, the kimi-webbridge skill) touches one of these subsystems, it is flagged inline and credited in full in the dedicated contributions section — not here.

### Architecture at a glance

```
 Telegram / Twilio / WhatsApp / Slack         macOS app · Chrome ext · web · CLI
        │ (signed webhooks /                        │ (edge JWT, aud=max-gateway)
        │  Socket Mode for Slack)                   │
        ▼                                           ▼
 ┌──────────────────────────────────────────────────────────────────┐
 │  GATEWAY  (Bun/TS, single public ingress)                         │
 │  · per-provider webhook signature validation + dedup              │
 │  · HS256 edge-JWT auth, scope profiles, policy-epoch revocation   │
 │  · edge→daemon token exchange (60s aud=max-daemon, subject→self)  │
 │  · per-IP auth-failure rate limiter, circuit breaker, retries     │
 │  · gateway-owned SQLite (auth/trust/contacts), risk classifiers   │
 │  · Velay outbound relay for NAT'd / self-hosted instances         │
 └──────────────────────────────────────────────────────────────────┘
        │ internal HTTP transport only (CI-guarded: no direct runtime URLs)
        ▼
   Assistant daemon (never faces the internet directly)
```

The daemon-is-off-the-internet invariant is enforced in CI by `gateway-only-guard.test.ts`, which bans direct runtime URLs anywhere in the codebase, so the gateway stays the single application-level ingress.

### Gateway service

A Bun/TypeScript edge service that is the assistant's only public face. It authenticates clients, validates and routes inbound channel webhooks, and brokers every call to the daemon over an internal HTTP transport.

#### Inbound channels & webhook ingress

Each provider webhook is signature-validated with the provider's own scheme before anything else runs:

| Channel | Validation mechanism | Path reference |
|---|---|---|
| Telegram | timing-safe compare of `x-telegram-bot-api-secret-token` | `gateway/src/telegram/verify.ts:3-17` |
| Twilio | HMAC-SHA1 over sorted params | `gateway/src/twilio/verify.ts` |
| WhatsApp / Meta | HMAC-SHA256 (`x-hub-signature-256`) | `gateway/src/whatsapp/verify.ts:6-30` |
| Slack | **not** inbound-signature-validated — runs **outbound Socket Mode** | dials `slack.com/api/apps.connections.open` with an app token, holds a WebSocket |

Slack is the deliberate exception: because the gateway dials *out* and holds the WebSocket, there is no inbound request to sign, so Slack events bypass the signature path entirely.

The ingress pipeline is **validate → dedup → route → intercept → forward** (`gateway/src/handlers/handle-inbound.ts:58-197`):

- **Dedup.** Telegram `update_id`s pass through a TTL cache (5-min TTL, 10k-entry cap) backed by a **monotonic high-water-mark**: any id at or below the mark is rejected *permanently*, closing the replay window that would otherwise reopen once a TTL entry is evicted (`gateway/src/dedup-cache.ts:31-66`).
- **Route.** `resolveAssistant` matches in priority order: explicit conversation-id route → explicit actor-id route → unmapped-policy default (or reject). Twilio inbound additionally reverse-maps the `To` phone number to an assistant via the config-file cache (`gateway/src/routing/resolve-assistant.ts:8-79`).
- **Intercept.** A text-verification intercept runs *before* forwarding, so the assistant never sees verification codes; `/new` resets the conversation.
- **Forward.** Traffic is forwarded to the runtime carrying a freshly minted `gateway_ingress_v1` token.

#### Client auth: single-header edge JWT with scope profiles

Client-facing `/v1` routes require an HS256 edge JWT (`aud=max-gateway`) signed with a 32-byte key from `ACTOR_TOKEN_SIGNING_KEY` or generated to disk at `0o600`. `verifyToken` checks 3-part structure, a timing-safe HMAC, audience, expiry, and a **monotonic `policy_epoch`** — a stale epoch is rejected, giving **blocklist-free revocation** (`gateway/src/auth/token-service.ts:166-219`). Scope is enforced by `resolveScopeProfile()` against a fixed `PROFILE_SCOPES` map (`gateway/src/auth/scopes.ts:15-43`), and routes select one of six auth strategies — `none` / `edge` / `edge-scoped` / `edge-guardian` / `track-failures` / `custom` (`gateway/src/http/router.ts:112-161`).

Two auth subtleties worth calling out:

- The **platform-managed auth bypass requires BOTH** `DISABLE_HTTP_AUTH` **and** `IS_PLATFORM`; either alone leaves JWT validation on. Even when active, the gateway cross-checks vembda's `X-Max-User-Id` header against a locally-stored `platform_user_id`, so the header alone is never a free pass (`gateway/src/http/middleware/auth.ts:43-49,94-132`).
- The **guardian-refresh route uses the `custom` strategy** that validates signature/audience/policy-epoch but *allows expired tokens* — otherwise a client whose access JWT just expired could never obtain a new one (a deadlock) (`gateway/src/index.ts:821-852`; `gateway/src/auth/token-exchange.ts:38-43`).

#### Edge→daemon token exchange

After validating an `aud=max-gateway` edge token, the gateway mints a fresh **60-second** `aud=max-daemon` exchange token per request, rewriting the subject's assistant segment to `self` so only the gateway can prove origin to the daemon (`gateway/src/auth/token-exchange.ts:23-130`). Webhook-originated traffic uses `mintIngressToken` (`gateway_ingress_v1`); gateway-originated calls use `mintServiceToken` (`gateway_service_v1`). UI-page tokens get a 1-hour (3600s) TTL (`gateway/src/auth/token-exchange.ts:160-167`). Because only the gateway holds the signing key, the daemon can trust caller origin without holding any secret itself.

#### Resilience: rate limiter, circuit breaker, retries

- **Per-IP auth-failure sliding window.** `AuthRateLimiter` blocks an IP after **10 failed auths in 60s** (429 + `Retry-After: 60`), bounded at **50,000** tracked IPs with stale-eviction + oldest-drop. Loopback peers are exempt and cleared on each check. It applies *only* to `/integrations/status`, `/webhooks/oauth/callback`, and `/v1/*` — it is **not** generic request throttling (`gateway/src/auth-rate-limiter.ts:4-61`; `gateway/src/http/middleware/rate-limit.ts:27-63`).
- **Module-global circuit breaker.** A single process-wide breaker (CLOSED/OPEN/HALF_OPEN) fronts every runtime call — forward, attachment, Twilio, OAuth-callback. **5 consecutive failures** open it for a **30s** cooldown, then exactly **one HALF_OPEN probe with retries disabled** decides re-close or re-open (cautious recovery, no thundering herd). **4xx is treated as success** (a client error, not an outage), so client mistakes never trip the breaker. Open state throws `CircuitBreakerOpenError` → **503 + Retry-After** so Telegram-style transports don't hot-retry (`gateway/src/runtime/client.ts:31-115,234-319`).
- **Forward retries.** 2 attempts, `500ms * 2^(n-1)` backoff, **30s** per-request timeout (`gateway/src/runtime/client.ts:259-275`; `gateway/src/config.ts:172-175`).

#### Gateway-owned SQLite (the auth/trust boundary)

Drizzle-declared tables (created via `CREATE TABLE IF NOT EXISTS`) hold the security boundary: `contacts`/`contact_channels`, `ingress_invites`, `actor_token_records` + `actor_refresh_token_records` (token-hash + hashed-device-id, **family-id rotation**, absolute + inactivity expiry), `trust_rules` (tool/pattern/risk/origin), channel verification sessions (defaults `codeDigits=6`, `maxAttempts=3`), guardian rate limits, and a denial-reply log (`gateway/src/db/schema.ts:76-339`). In Docker mode these — plus `trust.json` and the token signing key — live on a **gateway-only security volume the daemon cannot touch**.

#### Risk classification → auto-approve thresholds

Data-driven classifiers map a tool invocation to a risk level, feeding the interactive/autonomous/headless auto-approve thresholds stored in the DB:

- `bashRiskClassifier` parses commands against a default command registry + user trust rules (cached compiled regexes, escalate-one helper) (`gateway/src/risk/bash-risk-classifier.ts:38-68`).
- web classifier rules: `web_search` Low, `web_fetch` High when `allowPrivateNetwork`, `network_request` Medium (`gateway/src/risk/web-risk-classifier.ts:36-50`).

#### Feature-flag store + adaptive platform sync

Local overrides are file-backed (`feature-flags.json`, atomic temp+rename at `0o600`, module-cached). `RemoteFeatureFlagSync` polls the platform every **5 min** (10s initial, doubling backoff on failure, snap-to-steady on success; 10s fetch timeout), pausing entirely when credentials are absent and resuming on credential-cache invalidation (`gateway/src/remote-feature-flag-sync.ts:16-23,176-211`). A filter **drops remote `false` for any flag the local registry marks `defaultEnabled:true`** — defeating the platform's blanket-deny posture, which would otherwise silently turn off shipped GA features for every user (`gateway/src/remote-feature-flag-sync.ts:367-384`).

#### Velay outbound relay (NAT'd / self-hosted reach)

For instances behind NAT, the gateway dials *out* to Velay over a WebSocket (subprotocol `velay-tunnel-v1`, `Api-Key` auth), declaring an allowed-paths header. Velay proxies inbound HTTP/WS frames back through the tunnel to the gateway's loopback; a zod-validated frame protocol carries `http_request`/`response` and `websocket_*` frames. Heartbeat every **30s**, **60s** read-timeout, reconnect backoff `500ms → 30s` with 0.5 jitter ratio (`gateway/src/velay/client.ts:29-34,224-243`; `gateway/src/velay/protocol.ts:5-99`). Velay enforces a **4-entry path allowlist** — `^/webhooks/`, `^/v1/audio/`, `^/v1/live-voice$`, `^/v1/stt/stream$` — and an `allowed-paths` test enforces symmetry between that list and the gateway's actual public route surface (`gateway/src/velay/allowed-paths.ts:32-53`).

#### Twilio TwiML rewriting (keeping secrets out of the daemon)

The daemon emits sentinel placeholders for the relay token and public WSS URL; on the way back to Twilio, the gateway substitutes a **freshly minted relay JWT + the real public URL**, so neither the signing key nor the ingress URL ever lives in the daemon (`gateway/src/runtime/client.ts:500,586-605`).

#### Two corrections to common doc framing

- The gateway is **not** demonstrably the TLS-terminating front door: TLS terminates upstream at vembda in platform mode, or at the Velay edge with plaintext frames proxied to the gateway's loopback.
- Guardian-init auth is **deliberately not fail-closed**: three deployment modes (platform vembda-fronted; Docker bootstrap-secret header; bare-metal lockfile + loopback) use different mechanisms, and the docs explicitly warn scanners that managed mode reaching the handler without a bootstrap secret is the *expected* production state (`gateway/CLAUDE.md`).

### macOS Swift client

An AppKit-hosted SwiftUI app — the native Vellum/Max assistant — running as a menu-bar agent over a chat-centric main window with pop-out thread windows, conversation tabs/CRUD, and reconnect-time conversation restoration.

#### Service ownership: one DI container, no ambient globals

All long-lived state is owned by a single `AppServices` dependency-injection container created **once** in `AppDelegate` (connection manager, surface manager, secret/contact prompt managers, zoom, feature-flag and bookmark stores, ACP session store, settings, disk-pressure store). `AppDelegate` exposes only forwarding computed accessors — e.g. `var connectionManager: GatewayConnectionManager { services.connectionManager }` — so ownership stays in the container rather than scattered singletons (`clients/macos/vellum-assistant/App/AppDelegate.swift:82-119`). Tool-confirmation handling lives on `AppDelegate` as `toolConfirmationNotificationService`.

#### Transport: stateless HTTP client with dual route modes, plus SSE

`GatewayHTTPClient` is a stateless enum of static `get`/`post`/`delete(path:timeout:)` methods. `resolveConnection()` returns a `ConnectionInfo` carrying an `isManaged` flag (`clients/shared/Network/GatewayHTTPClient.swift:668-698`):

- **local / remote assistants** → `Authorization: Bearer` with flat `/v1/` paths.
- **platform-managed assistants** → `X-Session-Token` header with an injected `assistants/{assistantId}/` path prefix.

A long-lived `EventStreamClient` holds the daemon's SSE stream and fans events out to stores/managers (`clients/shared/Network/EventStreamClient.swift`).

#### Reconnection: two independent backoff regimes

| Concern | Behavior | Reference |
|---|---|---|
| Gateway health check | every **15s** | `GatewayConnectionManager.swift:94` |
| Gateway reconnect | fixed schedule `[3,5,10,15]s`, clamped at last entry | `:853` |
| Auto-wake / re-pair | 60s auto-wake cooldown; 90s re-pair timeout | `:797,:722` |
| Stale-loop guard | `reconnectionGeneration` counter invalidates old loops | `:867` |
| SSE reconnect | starts **1.0s**, doubles to **30.0s** cap, **resets to 1.0s on success** | `EventStreamClient.swift:129-130,710,717` |

#### ACP subagent session management

`ACPClient` (stateless enum over `GatewayHTTPClient`) exposes `listSessions`/`cancel`/`steer`/`delete`/`clearCompleted`, treating **404 as `success(false)`** ("definitely not running") and **409 as a real conflict** (still active); timeouts are 15s for list/clear and 10s for cancel/steer/delete (`clients/shared/Network/ACPClient.swift:46-159`). The observable `ACPSessionStore` keys per-session view-models by the **daemon UUID `state.id`** — *not* `acpSessionId`, which gets overwritten with the protocol handle once `createSession` resolves — uses identity (pointer) equality for stable SwiftUI `NavigationStack` paths, buffers orphaned updates that arrive before their spawn event, and caps each session log at **500 events** (oldest dropped first) (`clients/shared/Network/ACPSessionStore.swift:22-103`).

Wire decoders are **version-skew tolerant**: `Status`/`StopReason`/`UpdateType` all fall back to `.unknown` on unrecognized literals (`self = Status(rawValue: raw) ?? .unknown`), so a newer daemon adding a value never silently drops a session update (`clients/shared/Network/ACPMessages.swift:46-84,160-166`). *The kimi-agent stop reasons `max_turns`/`aborted`/`timeout` are explicit cases here, added by the author — see contributions section.*

#### On-host computer use with a client-side safety verifier

`HostCuExecutor` runs daemon-issued `host_cu_request` actions locally: `ActionExecutor` injects `CGEvent`s (click / type-via-clipboard-paste / key / scroll), `AccessibilityTree` walks `AXUIElement`, and `ScreenCapture` grabs a frame downscaled to **1280×720 / JPEG quality 0.6** via ImageIO (`clients/macos/vellum-assistant/ComputerUse/ScreenCapture.swift:40-95`). Crucially, safety is enforced **client-side in pure Swift, ahead of any CGEvent injection** by `ActionVerifier`: a **50-step cap**, **sliding-window loop detection** over the last **10 actions** (3 consecutive identical, or repeated 2–4-length cycles), plus credit-card/SSN/password and system-menu blocks and confirmation prompts for destructive key combos and form submits (`clients/macos/vellum-assistant/ComputerUse/ActionVerifier.swift:14,97,102-125`). Results post back via `host_cu_result`.

#### Screen recording

The standalone `ScreenRecorder` records via ScreenCaptureKit at **30fps** and steps down an **H.264 encode-config fallback ladder** — primary H.264 → 2×-downscaled H.264 → HEVC → 1280×720 H.264 safe fallback — on encoder failure (`clients/macos/vellum-assistant/Recording/ScreenRecorder.swift:338-560`). Recordings attach to the conversation as file-backed attachments.

#### Voice: live WebSocket with turn-based fallback

`LiveVoiceChannelClient` opens an authenticated WebSocket to `/v1/live-voice` (`pcm16kMono`), streaming mic PCM as binary frames plus `ptt_release`/`interrupt`/`end` control frames, decoding `ready`/`stt_*`/`assistant_text_delta`/`tts_*` frames. `VoiceModeManager` selects the live path **only** when the conversation has an id, the connection is healthy, and no tool prompt is pending — otherwise it falls back **once per session** to the standard turn-based Apple-Speech STT path, keeping the approval UX unchanged (`clients/shared/Network/LiveVoiceChannelClient.swift`).

#### WKWebView surfaces with JS bridge + hardening

`ui_surface_show` messages route through `SurfaceManager` to either a floating `NSPanel` (inline) or a full-window WKWebView workspace. `DynamicPageSurfaceView` injects three document-start `WKUserScript`s — design-system CSS, widget JS, and a `window.vellum` bridge exposing `sendAction`/`confirm`/`data` — and isolates each app behind a `https://*.vellum.local` origin via `VellumAppSchemeHandler`. Inline media webviews (`InlineVideoWebView.swift:37-151`) harden further with a non-persistent data store, per-provider host allowlists (YouTube/Vimeo/Loom), programmatic-navigation-only first loads (user-initiated navigations bounce to the system browser), and popup blocking (`createWebViewWith` returns nil).

> **Doc-drift note (corrected here):** older docs/prose reference `window.max` / `MaxAppSchemeHandler` / `max.local` and route-mode names like `runtimeFlat` / `platformAssistantProxy` / `/v1/messages`. Those reflect the older max-assistant tree and are **wrong for the live vellum-assistant tree** — the running identifiers are `window.vellum` / `VellumAppSchemeHandler` / `*.vellum.local`, and the route literals exist only in `ARCHITECTURE.md` prose, not in code. Documented JWT-refresh details (5-min check, refresh at 80% of TTL) are likewise prose-only; the client merely stores and compares a server-computed `refreshAfter` timestamp (`clients/shared/App/Auth/ActorTokenManager.swift:80-143`).

### Web shells, Chrome extension & CLI

Three (really four) independent front-ends. Two of the web surfaces are explicit **pre-product scaffolds**; the substantial browser surface is the Chrome extension; the `max` CLI is the provisioning/lifecycle tool.

#### Two web shells, both scaffolds

- **`apps/web`** — a Vite + React Router v7 SPA in **library/data-router mode** (`createBrowserRouter` + `RouterProvider`) with placeholder `conversations`/`library`/`settings` routes and a typed `RuntimeAuthAdapter` seam (`ensureSession`/`getAuthHeader`) whose modes are `hosted | local | self-hosted | none`. Library mode was chosen over framework mode as the more recognizable, fewer-conventions SPA shape for open-source contributors. Its README states *"No application code lives here yet"* and that feature PRs are redirected (`apps/web/README.md:3-14`).
- **`clients/web`** — a separate `Bun.serve` server (default **127.0.0.1:3000**) that builds a self-contained ESM bundle from `bundle.tsx` via `Bun.build`, memoizes it in a module-level `BundleCache` on first `/bundle.js` request, serves a `no-store` placeholder shell at `/`, and serves the bundle with `Access-Control-Allow-Origin: *` so a cross-origin platform host can `import()` it (`clients/web/src/server.ts:31-101`).

The real chat UI lives in a separate non-public repo pending migration.

#### Chrome extension (MV3) — the substantial browser surface

The MV3 service worker (`clients/chrome-extension/background/worker.ts`) opens an SSE stream and drives an explicit **7-state health machine** (`paused`/`connecting`/`connected`/`reconnecting`/`auth_required`/`assistant_gone`/`error`):

| Connection mode | Stream | Auth |
|---|---|---|
| self-hosted | gateway `/v1/events` (default `http://127.0.0.1:7830`) | `POST /v1/pair` JWT |
| max-cloud | `/v1/assistants/{id}/events` | WorkOS session |

It keeps alive via a `chrome.alarms` keepalive (period **0.5 min ≈ 30s**), reconnects with exponential backoff (**1s base, doubling, 30s cap**), and on **404 re-fetches the assistant list and auto-switches to a sole assistant** (`sse-connection.ts:19-278`; `worker.ts:602-692,906`). Minimum Chrome version is 120, but flat sessions require 125+ (`manifest.json:5`).

`host_browser_request` envelopes are routed into a **CDP proxy over `chrome.debugger`** (`cdp-proxy.ts:178-302`). For **Chrome 125+ flat sessions**, the child `sessionId` is routed via the `DebuggerSession` *target argument* (not `params`); `send()` always resolves a `CdpResultFrame`, mapping errors to JSON-RPC `-32000`/`-32602`. The `host-browser-dispatcher.ts` guarantees **deterministic cancellation** via per-call `AbortController`s plus a `cancelledRequestIds` set with identity-guarded cleanup, so an overlap-retry across a relay reconnect can't suppress a legitimate result or leak a ghost completion after the daemon already returned `Aborted`; it also dedupes `chrome.debugger.attach` via an `attachedTargets` cache evicted on detach / "not attached" / "target closed". Synthetic `Max.findTab`/`Max.attach`/`Max.detach` methods (plus `Page.navigate` recovery) handle **target resolution only** — the automation primitives (`navigate`/`evaluate`/`screenshot`/`printToPDF`, etc.) are ordinary CDP commands forwarded through the proxy. Results are POSTed back over HTTP to `/v1/host-browser-result`.

> **Doc-drift note (corrected here):** the README calls this a *"WebSocket relay"* (`README.md:3`) and dispatcher docstrings say "over the existing browser-relay WebSocket", but the implementation is **SSE inbound + HTTP POST outbound** — the code is authoritative. Similarly, the build-default environment falls back to `"production"` in code (`extension-environment.ts:91`) while the README precedence table says `"dev"`; the bundle-time `--define` injection sets the right value per subcommand, so the fallback only bites unbundled loads.

#### `max` CLI

The Bun-built CLI (`cli/src/index.ts:32-191`) dispatches `argv[0]` through a `commands` record (**26 command names backed by 24 module files** — `login.ts` contributes `login`/`logout`/`whoami`), stripping global `--no-color`/`--plain`/`--version`/`--help` first. With no command, it probes for a healthy running assistant (`resolveAssistant` + `checkHealth`) and launches the TUI client, else prints help.

It provisions assistants locally or on cloud VMs — `max hatch --remote gcp` creates a GCP **e2-standard-4 (4 vCPU / 16 GB)** Compute Engine instance with a metadata startup script (`cli/src/lib/gcp.ts:296`) — and manages lifecycle: `wake`/`sleep`/`ps`, `terminal`/`exec`/`ssh`, `backup`/`restore`/`teleport`, `retire`/`recover`, `login`/`whoami`. `retire` reads the saved `cloud` field from the lockfile to dispatch teardown across `gcp`/`aws`/`docker`/`custom`.

The CLI **deliberately does not declare `clients/web` as a dependency**: `--interface web` walks up the tree to find the sibling `clients/web` package on disk and spawns its `local` script, keeping the published CLI decoupled from the unpublished web package (`cli/src/commands/client.ts:277-318`).

Default service ports & probe constants (`cli/src/lib/constants.ts:14-17`):

| Constant | Value |
|---|---|
| Gateway port | 7830 |
| Daemon port | 7821 |
| Qdrant port | 6333 |
| CES port | 8090 |
| Health probe timeout | 1500ms (secondary fetches 5000ms) |
| Job polling | 2000ms interval, 1-hour cap |
