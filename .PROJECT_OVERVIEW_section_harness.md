## The agentic harness & multi-provider model layer

This is the platform's inference engine: the loop that turns a model response into tool calls and back, the context-defense machinery that keeps a 200k-token window from overflowing mid-conversation, and the decorator-chained provider abstraction that fronts Anthropic, OpenAI, Gemini, Ollama, Fireworks, and OpenRouter behind one `sendMessage()` signature. Everything in this section is Vellum/Max platform code. Where the author extended a platform mechanism (the kimi-agent provider, the claude-subscription bridge hardening), it's flagged inline and credited in full in the dedicated contributions section — not here.

### The agent loop: `stop_reason`-driven, no hard step cap

The core is `AgentLoop.run()` (`assistant/src/agent/loop.ts:416-1044`). It is a `while(true)` cycle with **no iteration counter gating it** — by design. Each turn:

1. `provider.sendMessage(messages, tools, systemPrompt, options)` streams a response.
2. The response content is filtered for `type: 'tool_use'` blocks.
3. If there are none (or no tool executor is wired), the loop **breaks** — that is the primary termination condition.
4. Otherwise it emits `tool_use` events, runs **all** tools concurrently via `Promise.all` raced against an `AbortSignal`, pushes the `tool_result` blocks back as a single user message, and loops.

Termination is driven by model behavior plus three explicit escapes — a tool requesting `yieldToUser`, the abort signal firing, or an orchestrator `onCheckpoint` returning `'yield'` (`loop.ts:1081-1130,1258-1261`). The only numeric caps in the loop body are an empty-response retry (1) and a consecutive-tool-error nudge cap (3); neither bounds useful work. A `minTurnIntervalMs = 150` throttle (`loop.ts:156,512-518`) prevents spin when tools return instantly.

| Loop constant | Value | Location |
|---|---|---|
| `MAX_EMPTY_RESPONSE_RETRIES` | 1 | `loop.ts:160` |
| `MAX_CONSECUTIVE_ERROR_NUDGES` | 3 | `loop.ts:159` |
| `minTurnIntervalMs` | 150 ms | `loop.ts:156` |
| `MAX_AX_TREES_IN_HISTORY` | 2 | `loop.ts:1374` |
| Default `maxTokens` (output) | 64000 | `loop.ts:154` |
| Effort: standalone-fallback vs production | `'high'` (test) vs `'max'` (schema) | `loop.ts:155` / `config/schemas/llm.ts:301` |

**Empty-response nudge gate.** If a turn produces no visible text and no `tool_use` *after* tool work already happened, the `emptyResponse` plugin pipeline returns `'nudge'` exactly once (`MAX_EMPTY_RESPONSE_RETRIES = 1`), pushing a `<system_notice>` user message; otherwise `'accept'` (`loop.ts:934-1016`, `plugins/defaults/empty-response.ts:83-86`). Bridged tool activity only opens this gate when the provider sets `supportsEmptyTurnNudge` per call — a flag only the session-resuming bridge providers assert, because re-nudging is only safe when the inner session can be resumed rather than re-executed.

**Tool-error retry nudge.** When any tool result is an error, `consecutiveErrorTurns` increments and the `toolError` pipeline appends a corrective text nudge to the result blocks. After `MAX_CONSECUTIVE_ERROR_NUDGES = 3` consecutive error turns, nudging is suppressed so an unrecoverable error can't spin the loop (`loop.ts:159,1271-1310`).

### Context hygiene per turn: images, AX-trees, truncation, secrets

Around every turn the loop applies four independent context-shaping passes so prompt size — and therefore time-to-first-token — doesn't grow linearly with step count, which matters acutely in computer-use sessions:

- **Old-image stripping** (`stripOldImageBlocks()`, `loop.ts:1458-1514`): image content blocks are removed from every `tool_result` *except* the most recent tool-result user message, replaced with a text marker. Each screenshot is seen once on capture, then dropped.
- **AX-tree history compaction** (`compactAxTreeHistory()`, `loop.ts:1373-1456`): keeps only the most recent `MAX_AX_TREES_IN_HISTORY = 2` `<ax-tree>` blocks (counted per block, not per message) and replaces older ones with `<ax_tree_omitted />`.
- **Tool-result truncation** (`tool-result-truncation.ts:11-21,64-71`): `calculateMaxToolResultChars(window) = min(HARD_MAX_TOOL_RESULT_CHARS = 400000, floor(window * 0.3 * 4))` — a hard 400k-char ceiling, otherwise ~30% of the window at ~4 chars/token — preserving at least `MIN_KEEP_CHARS = 2000`. The same cap is applied to bridged tool results before they enter an SDK's context (`loop.ts:720-724,1170-1216`).
- **Sensitive-output placeholders** (`loop.ts:866-883,1141-1150`): real secret values are substituted **only** into streamed `text_delta` events (chunk-safe via a pending buffer). Placeholders — not real values — persist in both the provider message history and the persisted store, so the model never re-sees a secret on a later turn or after a DB reload. The same merge runs at the bridge boundary for SDK-driven providers (`loop.ts:664-668`).

Image payloads are additionally downscaled for transport before they ever reach the loop: `optimizeImageForTransport()` (`agent/image-optimize.ts:19-27,140-177`) shrinks any image with a side > 1568px (or > 300KB when dimensions can't be parsed) to JPEG quality 80 via macOS `sips`, behind a content-addressed disk cache capped at 500 entries. 1568px matches Anthropic's server-side scaling, so this is zero quality loss and avoids 413s.

### Two-tier context defense

A 200k-token window (`maxInputTokens`, `config/schemas/llm.ts:200`) is defended both proactively and reactively.

**Proactive — mid-loop pre-emptive yield.** The orchestrator's `onCheckpoint` (`daemon/conversation-agent-loop.ts:2060-2099`) estimates prompt tokens after each tool turn; if the estimate exceeds `preflightBudget * 0.85` it returns `'yield'`, the loop breaks, compaction runs, and the loop re-enters — compacting *before* the provider can reject the prompt.

**Reactive — `ContextWindowManager`.** `shouldCompact` / `maybeCompact` (`context/window-manager.ts:401-481`) summarize older user turns via a bundled `compact.md` prompt (with an inline fallback) once the estimate reaches `floor(maxInputTokens * compactThreshold)`, keeping a recent-turns tail and re-targeting toward `targetBudgetRatio`. Tool results inside the compactable slice are pre-truncated to `COMPACTION_TOOL_RESULT_MAX_CHARS = 6000` (`window-manager.ts:31`).

| Context-defense constant | Value → effect | Location |
|---|---|---|
| `maxInputTokens` | 200000 tokens | `llm.ts:200` |
| `compactThreshold` | 0.8 → compact at ~160k | `llm.ts:213` |
| `targetBudgetRatio` | 0.3 → compact toward ~60k | `llm.ts:212` |
| `summaryBudgetRatio` | 0.05 | `llm.ts:214` |
| Mid-loop yield threshold | `preflightBudget * 0.85` | `conversation-agent-loop.ts:2063` |
| `overflowRecovery.maxAttempts` | 3 | `llm.ts:181` |
| Compaction circuit breaker | opens after 3 consecutive failures; 1-hour (3,600,000 ms) cooldown | `plugins/defaults/circuit-breaker.ts:42-48` |

The overflow-recovery loop is bounded at 3 attempts, and a circuit breaker suspends compaction for one hour after 3 consecutive summarizer failures — so a failing summarizer thrashing against an oversized prompt can't lock up the conversation.

### Sub-agents: role-scoped, depth-capped

`SubagentManager` (`subagent/manager.ts:141-149`) spawns a child `Conversation` per role — `general` / `researcher` / `coder` / `planner` — each with its own `allowedTools` allowlist (`subagent/types.ts:95-164`). `SUBAGENT_LIMITS.maxDepth = 1` (`subagent/types.ts:95-98`) blocks a sub-agent from spawning nested sub-agents. Concurrency here is depth-only: there is no fan-out count cap in the manager. Terminal sub-agent metadata is retained 30 minutes and swept every 5 (`manager.ts:38-40`).

### SDK-driven bridge providers

Two providers dispatch Max tools *inside their own SDK loop* via a `toolBridge` closure (`loop.ts:596-700`) rather than through the main loop's executor:

- **kimi-agent** *(author-built — see contributions section)*: a per-conversation kimi-cli session resume, a `StepBegin` step-guard backstop, a `MAX_CONCURRENT_CALLS = 4` concurrency cap with excess queued, and a `streamTimeoutMs` wall-clock guard defaulting to 1,800,000 ms (30 min) (`providers/kimi-agent/client.ts:132,190,548`).
- **claude-subscription** *(platform bridge — hardened by the author)*: the platform was extended to throw a typed `ContextOverflowError` on a `'Prompt is too long'` response so `RetryProvider` routes straight to compaction instead of retrying a prompt that can only fail again (`providers/claude-subscription/client.ts:139-144,624-640`).

The bridge plumbing inside `loop.ts` — the `toolBridge` closure, `bridged_tool_committed` / `bridged_tool_result` events, `supportsEmptyTurnNudge`, the `bridgedToolCalls` counter — is pre-existing platform code.

### One honest, citable defect: comment-vs-code drift in the `maxTurns` backstops

Both SDK recursion backstops now read `100000`, while their surrounding comments still describe the old, much smaller intent — the code contradicts its own stated design:

- `providers/kimi-agent/client.ts:190` sets `const MAX_TURNS = 100000`, but the comment directly above (`client.ts:184-188`) insists on keeping "the host cap **UNDER 100** … so OUR graceful '[Stopped early …]' note fires before the CLI's" 100-step limit. (This change is working-tree-only / uncommitted, consistent with the documented rebuild-required state.)
- `providers/claude-subscription/client.ts:466` sets `maxTurns: 100000` (committed in `04762ca6`, raising it from 50), while the adjacent comment (`client.ts:462-465`) still reads "50 = the safety ceiling the provider tests pin."
- The kimi per-call schema clamp is `int 1..100000` (`config/schemas/llm.ts:313,335`), yet the code comment near `client.ts:297-300` claims a "clamp to 200."

In practice the *effective* per-call bound for kimi-agent is `config.maxTurns` enforced by the `StepBegin` guard, not the 100000 constant — but the constants and their comments diverge, and a reader trusting the comments would be wrong about all three.

### Multi-provider model layer

Beneath the loop sits a uniform `Provider.sendMessage(messages, tools, systemPrompt, options)` interface implemented by every transport and every decorator. The platform fronts Anthropic, OpenAI, Gemini, Ollama, Fireworks, and OpenRouter (plus the separately-credited Kimi and claude-subscription bridges) behind it.

**Single source of truth + boot-time drift guards.** One `PROVIDER_CATALOG` (`providers/model-catalog.ts`) and a catalog-keyed `ADAPTER_FACTORIES` table (`providers/inference/adapter-factory.ts:57-149`) drive everything. A self-executing `PROVIDER_CATALOG_FACTORY_PARITY` block throws at module load if catalog ids and factory ids diverge (a missing or orphan factory). A second guard in `model-intents.ts:64-73` throws if any model-intent (`balanced` / `latency` / `quality` / `vision`) maps to a model id absent from the catalog. Both invariants fail at boot, not at user-call time.

**Decorator chain.** At boot, `registry.registerProvider` resolves credentials in a waterfall — user API key → managed platform proxy → keyless/CLI-login (Ollama and the bridge providers) — then wraps the raw adapter as `UsageTrackingProvider(RetryProvider(adapter))` (`registry.ts:35-37,208-213`). Per-conversation construction (`daemon/conversation-store.ts:239-247`) layers `CallSiteRoutingProvider` **always** and `RateLimitProvider` **only when** `rateLimit.maxRequestsPerMinute > 0` (schema default 0 = disabled). Each wrapper implements the same signature and delegates inward.

**Retry with equal-jitter backoff.** `RetryProvider` (`providers/retry.ts:515-564`) loops `attempt = 0..DEFAULT_MAX_RETRIES` — up to 3 retries, 4 total attempts. `isRetryableError` returns true for 429/5xx, overloaded SSE messages, stream-corruption patterns, transport aborts, and ECONNRESET-class network errors. Delay is `min(server Retry-After or computeRetryDelay(attempt, base=1000ms), 60000ms)`; `computeRetryDelay` returns `cap/2 + random(0..cap/2)` (equal jitter) where `cap = base * 2^attempt` (`util/retry.ts:8-9,23-30`). Server-suggested `Retry-After` is honored but also capped at 60s.

**Context overflow is deterministically non-retryable.** `isRetryableError` checks `isContextOverflowError` *first* (`retry.ts:130-135`), before the generic 429/5xx branch — so an oversized prompt (which on Gemini/Vertex arrives as a 429 `RESOURCE_EXHAUSTED`) never burns a retry. `ContextOverflowError` (extends `ProviderError`) is constructed across every keyed transport: Anthropic (`detectAnthropicContextOverflow`, `anthropic/client.ts:39-59`), Gemini, the shared OpenAI-compat path (`context_length_exceeded` family), OpenAI Responses, and an OpenRouter rewrap.

**Per-provider config normalization.** `normalizeSendMessageOptions` (`retry.ts:174-422`) resolves the call-site config, then strips knobs the target provider rejects:

| Knob | Stripped for |
|---|---|
| `thinking` | non-Anthropic / non-OpenRouter |
| `effort` | non-Anthropic / OpenAI / OpenRouter / Fireworks |
| `speed` | non-Anthropic |
| `verbosity` | non-OpenAI |
| `openrouter.only` | non-OpenRouter |
| `maxTurns` | everything but kimi-agent |

It additionally drops `thinking` under Anthropic's wire constraints — forced `tool_choice` or `temperature != 1`. For OpenRouter, `thinking`-on-forced-tool-use is stripped only when the model id starts with `anthropic/` (`retry.ts:329-341`).

**Streaming deadline.** Every adapter streams under a linked `AbortController` deadline (`createStreamTimeout`, `stream-timeout.ts:9-39`) defaulting to 1,800,000 ms (30 min, via `timeouts.providerStreamTimeoutSec`). One nuance the survey over-generalized: **only the Anthropic adapter** passes the SDK a request timeout of `streamTimeoutMs + 60_000` (`anthropic/client.ts:745`) so the inner `AbortController` always fires first with a clearer message; the OpenAI/Responses adapters construct their SDK client with no request timeout and rely solely on the inner signal. Anthropic's catch-site disambiguates three failure modes the SDK collapses into `'Request was aborted'` with `status === undefined`: caller cancellation (tagged, never retried), inner stream-timeout (message rewritten to `'timed out after Xs'`, falls through to network classification), and transport abort (left verbatim, matched by an Anthropic-anchored regex and retried) (`retry.ts:75-128`, `anthropic/client.ts:1418-1531`).

**Per-call-site routing without rebuilding the transport.** `CallSiteRoutingProvider` (`call-site-routing.ts:52-167`) consults the resolved profile per call; when it names a `provider_connection` it resolves that connection's auth and delegates to its transport. The active routed provider name is stored in an `AsyncLocalStorage` rather than an instance field, so two concurrent `sendMessage` calls on the same provider instance (e.g. a main agent turn and a title-generation call) each report their own routed provider during streaming without clobbering each other. Hard config errors throw `ConnectionResolutionError`; soft credential failures fall back to the default provider.

**Sliding-window rate limiter.** `RateLimitProvider` (`ratelimit.ts:59-100`) is a request-log, not a token bucket: it prunes timestamps older than 60s in-place in a single O(n) pass over a shared array, and throws `RateLimitError` with a computed `retryAfterSec` once `count` reaches `maxRequestsPerMinute`. Disabled at the schema default of 0.

**OpenAI's two transports; OpenRouter's dual reasoning path.** OpenAI is split across `OpenAIResponsesProvider` (maps `effort → reasoning.effort`, `verbosity → text.verbosity` for GPT-5-series knobs) and the shared `OpenAIChatCompletionsProvider` reused by Ollama, Fireworks, and Kimi (maps `effort → reasoning_effort`, capped per provider) — one transport class, four vendors, parameterized by `baseURL` / `providerName` / `maxReasoningEffort` (`adapter-factory.ts:70-115`). Fireworks and Kimi both pin `maxReasoningEffort` to `'high'`, collapsing `xhigh` / `max`; OpenAI Responses caps `max → xhigh`. OpenRouter (`openrouter/client.ts:160-177`) translates internal `thinking` into its unified `reasoning` parameter for OpenAI-compat models but routes `anthropic/*` models through an internal `AnthropicProvider` so Anthropic-native thinking is forwarded intact.

**Usage attribution.** `UsageTrackingProvider` records a cost event per call site with structured pricing lookup; `X-Max-*` attribution headers ride along **only** on the managed-proxy surface (`forwardUsageAttributionHeaders = true`), i.e. only when the platform is the billing party (`registry.ts:97-114,208-213`, `retry.ts:430-466`).

|---|---|---|
| Stream timeout (inner deadline) | 1,800,000 ms (30 min) | `timeouts.ts:35-42` |
| Anthropic SDK request-timeout buffer | `streamTimeoutMs + 60,000 ms` | `anthropic/client.ts:745` |
| `DEFAULT_MAX_RETRIES` | 3 (→ 4 total attempts) | `util/retry.ts:8` |
| Retry base / formula | base 1000 ms; `cap/2 + random(0..cap/2)` | `util/retry.ts:9,23-30` |

A parallel speech-to-text provider family (Deepgram, OpenAI Whisper, Google Gemini Live, xAI realtime) lives under `speech-to-text/` with its own catalog and transports — outside this text-inference focus. The thin gated hooks for the author's own providers (the `maxTurns`-forwarding branch in `retry.ts`, `FLAG_GATED_PROVIDERS` registration gating in `registry.ts:137-153`) thread through these shared files but the providers themselves are credited separately.
